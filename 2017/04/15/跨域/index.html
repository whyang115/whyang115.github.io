跨域

发表于 2017-04-15 | 分类于 笔记总结 |

浏览器在访问其他域的资源时会因为同源策略的影响而失败,这便是跨域问题,针对跨域问题也有很多不同的方案各有其使用场景

什么是跨域

浏览器为了安全起见引入了 同源策略 来提高安全性,所谓同源策略即同域名、同端口、同协议的才能互相获取资源,具体限制由下图示

如果基础域名相同、协议和端口也相同,便可以通过基础域名的方式来通信

window.name

window.name这个全局属性主要是用来获取和设置窗口名称的,但是结合iframe也可以实现跨域,因为每个iframe都有外层包裹它的window对象,而这个window是最外层的对象,所以window.name属性可以共享

  1. message: 要传递的信息对象,只支持字符串形式

jsonp实现跨域

由于同源策略不会阻止动态脚本的插入,由此衍生出了一种很常用的跨域方式: JSONP(JOSN with Padding)即JSON填充,意为在数据中填充上回调函数

利用动态脚本的src属性,发送了一个http://www.b.com/getdata?callback=show的GET请求,b.com页面接收的这个请求时若不支持JSONP,会正常返回一个数据,假设为{msg: 'I'm message'},而通过JSONP,b.com会接受这个callback参数,利用这个参数值包装返回的数据show({msg: 'I'm message'}),而a.com正好有一个show函数,如此一来,当远程数据返回时,这个show函数即会自动执行;
JQuery中也有封装好的JSONP函数,使用$.getJSON方法来请求跨域数据

还可利用jquery中的$.ajax方法,只需要指定dataType为jsonp即可

JSONP方式虽然因为兼容性较好的得以在低版本浏览器中用途广泛,但由于其自身的局限性,它只支持GET请求,对要求安全性的跨域需要谨慎使用,另外他没有错误处理机制,一旦发生错误,浏览器会以静默失败的方式处理。在现代浏览器中可以使用CORS方法

CORS方法

跨域资源共享(Cross-origin resource sharing) w3c制定的一个标准用来在现代浏览器中实现跨域请求
CORS的主要原理是通过一系列的HTTP头信息来实现服务器和客户端的通信,所以要实现CORS,需要对服务器端和客户端都做好相应的配置
在CORS标准中请求分为两类: 简单请求和非简单请求

简单请求

满足以下条件为简单请求:

  1. 请求方法为: GET、HEAD、POST
  2. 对于请求头字段有严格要求,一般情况下不会超出以下字段:
    • Accept
    • Accept-Language
    • Content-Language
    • Content-Type
      3.当发起POST请求时,只允许Content-Type为application/x-www-form-urlencoded,multipart/form-data,text/plain

简单请求中服务器和客户端之间的通信只是进行简单的交换,如图:

浏览器发送一个带有Origin字段的HTTP请求头,用于表明请求来源。服务器的Access-Control-Allow-Origin响应头表明服务器允许哪些源得的访问,*表示任意源.

非简单请求

对于非简单请求,在正式发送请求数据之前,浏览器会先发送一个带有OPTIONS方法的请求来确保该请求是安全的,这个请求又被称为‘预请求’(preflight)
浏览器与服务器间的通信如图示:

浏览器会在预请求中多发送两个字段Access-Control-Request-Method和Access-Control-Request-Headers,前者用于告知服务器实际请求所用的方法,后者用于告知服务器实际请求所携带的自定义请求首部字段,然后，服务器根据请求头的信息来判断是否允许该请求
对于非简单请求,服务器端可以设置几个相关字段:

  1. Access-Control-Allow-Methods: 用来限制允许的方法名
  2. Access-Control-Allow-Header: 用来限制允许的自定义字段名
  3. Access-Control-Allow-Credentials: 用来表明服务器是否允许credentials标志为true的场景
  4. Access-Control-Max-Age: 用来表明预请求响应的有效时间
  5. Access-Control-Expose-Headers: 用来指定服务器端允许的首部字段集合
